Eindreflectie

Tijdens de afgelopen zes maanden heb ik veel geleerd over webdevelopment, zowel op technisch als op persoonlijk vlak. In deze reflectie blik ik terug op de belangrijkste inzichten uit de verschillende vakken en de meesterproef, en kijk ik vooruit naar hoe ik deze ervaringen wil meenemen tijdens mijn stage en verdere ontwikkeling.

Geschreven door Kerr

Sprint 0

Tijdens sprint 0 was het eerste moment dat ik realiseerde dat ik nog veel te leren had. Ik was bezig met het maken van mijn opdracht toen ik een kleurenthema moest maken. Ik vroeg toen aan Sanne of er in JavaScript iets van een `Color` klasse bestond in Javascript. Sanne vroeg me toen verbaasd waarvoor ik dit nodig had en dat dit gewoon in CSS kon.

Deze anekdote laat zien dat ik op dit punt vooral dacht vanuit mijn achtergrond als programmeur. Uiteraard was ik bekend met het scheiden van de UI en achterliggende code, maar in de praktijk had ik nog maar weinig gewerkt met het opbouwen van interfaces waardoor ik snel naar Javascript greep.

Ik denk dat dit ook een van de belangrijkste lessen is die ik heb geleerd tijdens de minor: er is meer mogelijk met HTML en CSS dan ik voorheen dacht. Het eerste blok vakken benadrukte dit nog eens.

Blok 1: CSS en Browser Tech

Het eerste blok kende twee vakken. *Browser Technologies* en *CSS to the Rescue* Tijdens het dit eerste vak ben ik aan de slag gegaan met webforms. Ter voorbereiding van de minor had ik wat courses gevolgd die ook webforms hadden genoemd en ik had er al een paar gebouwd, maar nog niet op dit niveau. Doordat bij dit vak ik voor een langere tijd aan de slag ging met het maken van 1 enkele form, heb ik echt de tijd gehad om veel kleine details rekening te houden waar ik voorheen niet aan gedacht zou hebben. Een van de belangrijkste principes die ik heb geleerd tijdens dit vak is het principe van *Progressive Enhancement*. Dit principe stelt dat een goed ontwerp blijft werken als alle geavanceerde functionaliteit wegvalt. Wellicht is het beter te stellen dat een ontwerp zo moet worden ontworpen dat primitive technologieën de basiswerking garanderen en dat geavanceerder technologie ervoor kan zorgen dat deze basiswerking wordt verbeterd.

In het geval van web development betekent dit dat een website zou moeten werken met enkel HTML, wat een robuuste, maar simpele technologie is. Vervolgens kan CSS ervoor zorgen dat deze HTML beter werkt door de website mooier te laten ogen, maar ook de gebruiker met styling een betere user experience te bieden. Zowel HTML als CSS zijn declaratief, wat betekent dat de programmeur aangeeft wat hij zou willen en dat de browser dit vervolgens regelt. JavaScript kan vervolgens gebruikt worden om de gebruikerservaring naar een volgend niveau te tillen. Javascript is een imperatieve taal, wat betekent dat de programmeur exact moet aangeven wat het programma stap voor stap moet doen. Dit maakt Javascript fragiel, omdat elk van deze stappen kan falen waardoor de rest van het programma ook niet werkt. Dit laat wederom het belang van progressive enhancement zien.

Ik denk dat dit ook een van de belangrijkste lessen is die ik heb geleerd tijdens de minor: er is meer mogelijk met HTML en CSS dan ik voorheen dacht. Het eerste blok vakken benadrukte dit nog eens.

Tegelijkertijd heb ik bij het vak CSS to the rescue mijn kennis over CSS flink verbreed. Voordat ik aan de minor begon was mijn kennis over CSS erg zwak. Ik had vrijwel niet gewerkt met verschillende selectoren. Tijdens dit vak mocht je geen classes gebruiken, waardoor je alles moest oplossen met selectoren. De twee selector die ik het meest praktisch vond is de `has()` selector, die het mogelijk maakt om de styling van een parent aan te passen op basis van de children van het element. Dit was voorheen niet mogelijk, maar met deze selector kan dit dus wel. Een andere selector die ik interessant vond om te gebruiken was de `not` selector, die werkt zoals een logische negatie.

Tijdens dit vak ben ik ook een super cool project tegengekomen, namelijk deze x86 CPU in CSS. Dit is een processor die volledig gemaakt is in CSS en in staat is een C programma uit te voeren. Ver boven mijn kunnen, maar ik vond dit een leuke samenloop van mijn interesse in webdevelopment en technische informatica.

Blok 2: API en Human Centered Design

Na een blok waarin ik vooral geleerd heb om anders te denken over HTML en CSS, verschoof de focus in blok 2 naar twee heel andere aspecten van web development: het werken met externe API's en het ontwerpen vanuit de gebruiker.

Voor het vak API heb ik Light Wave ontwikkeld, een applicatie die Spotify en Philips Hue combineert om de sfeer van een ruimte af te stemmen op de emotie van de gebruiker. Dit vak voelde in veel opzichten als een vervolg op mijn achtergrond als programmeur, maar dan toegepast op het web. Ik kreeg voor het eerst de kans om zelfstandig een volledige OAuth-flow te implementeren, en dat gaf me een veel dieper begrip van hoe authenticatie en het veilig afhandelen van tokens eigenlijk werkt.

Wat ik vooral leerzaam vond, was het samenspel tussen verschillende technologieën binnen één project: server-side endpoints in Astro, een externe database via Turso en Astro DB, en twee compleet losstaande API's die ik met elkaar moest laten samenwerken. Vooral het synchroniseren van twee systemen die niets van elkaar weten (Spotify en Hue) leerde me goed nadenken over foutafhandeling. Dit resulteerde uiteindelijk in een eigen `HttpError`-klasse. Ook het werken met `Promise.all()` en `Promise.allSettled()` gaf me een beter gevoel voor asynchrone code, en hoe je moet omgaan met meerdere taken die tegelijk kunnen slagen of falen.

Tijdens de herkansing heb ik daarnaast de gebruikerservaring flink verbeterd met herbruikbare Astro Components, layouts en View Transitions.

Waar API vooral over de techniek achter een applicatie ging, draaide Human Centered Design volledig om de gebruiker. Ik heb een prototype ontwikkeld van een chatapplicatie waarin het ontvangen van spraakberichten centraal stond, en heb dit door meerdere iteraties heen getest met Ihab, een blinde gebruiker die met een screenreader werkt.

Dit vak zette voor mij echt aan het denken. Zaken die ik als vanzelfsprekend beschouwde, zoals tegelijk luisteren naar audio en typen, bleken voor een screenreadergebruiker een serieuze uitdaging. Doordat ik telkens met dezelfde gebruiker testte, zag ik heel concreet het effect van mijn ontwerpkeuzes: van standaard audiobediening, naar toegankelijke custom controls met keyboard shortcuts, naar uiteindelijk een AI-samenvattingsfunctie voor langere berichten. Elke iteratie loste een probleem op dat ik zelf nooit had voorzien, wat me liet inzien hoe beperkt mijn eigen blik als ontwikkelaar eigenlijk is als ik niet direct met gebruikers samenwerk.

Wat dit vak voor mij extra waardevol maakte, in combinatie met API, is dat ik leerde dat toegankelijkheid en techniek niet los van elkaar staan. Het gebruik van `aria-keyshortcuts`, een goede semantische structuur en doordachte keyboardnavigatie zijn net zo goed technische beslissingen als het opzetten van een OAuth-flow, alleen dan met de gebruiker als uitgangspunt in plaats van de data of de API. Dit blok liet me zien dat een applicatie pas echt goed is als zowel de techniek onder de motorkap klopt, als de ervaring die de gebruiker ermee heeft.

Meesterproef

Als ik terugkijk op de vijf weken die ik aan het project heb gewerkt, valt me vooral op hoe erg mijn aanpak veranderde van "zo snel mogelijk iets werkends bouwen" naar "iets bouwen dat ook voor anderen te begrijpen en uit te breiden is". In week 1 en 2 lag de focus nog vooral op onderzoek en het neerzetten van een eerste werkende structuur: de keuze voor GSAP, het opzetten van een JSON-datastructuur voor animaties en de eerste versie van de AnimationBuilder. Dit ging me vrij natuurlijk af, omdat het aansloot bij mijn achtergrond als programmeur.

De echte les kwam in week 3. Ik was tevreden met de opsplitsing van de AnimationBuilder in een AnimationData- en AnimationPlayer-klasse, en met het dependency inversion-principe dat daaruit voortkwam, maar ik was te veel gefocust op wat ik zelf een goede architectuur vond. Pas tijdens de code review met Cyd werd duidelijk dat de rest van de groep niet meer kon volgen wat er in de code gebeurde, en dat dit de voortgang van het hele project in gevaar bracht. Door samen met Mila de code door te nemen, comments te schrijven en JSDoc toe te voegen, leerde ik dat het simpelweg uitleggen van je eigen code een van de beste manieren is om onnodige complexiteit eruit te halen.

Deze ervaring heeft ook mijn beeld van teamwork binnen een technisch project veranderd. Aan het begin van het project waren de rollen vooral organisatorisch (notulist, planner, contactpersoon), maar naarmate het project vorderde werd steeds duidelijker dat goede samenwerking op codeniveau minstens zo belangrijk is: niet alleen zorgen dat je eigen stuk werkt, maar ook dat je groepsgenoten het kunnen begrijpen, aanpassen en verder bouwen.

Uiteindelijk was het mooi om te zien waar dit alles toe leidde. Ondanks dat week 4 en de eerste helft van week 5 nog vrij hectisch waren met het afronden van save/load en undo/redo, en er tijdens de demo nog foutjes naar boven kwamen, was de overdracht met Marjolijn een fijne afsluiting. Dat zij ons meerdere keren bedankte en enthousiast was om de tool voor haar PD te gebruiken, gaf me het gevoel dat alle keuzes rondom architectuur, leesbaarheid en samenwerking uiteindelijk hebben bijgedragen aan een mooi eindproduct.

Vooruitblik

De minor heeft me vooral geleerd dat een goede architectuur pas echt waardevol is als anderen hem ook kunnen volgen. Tijdens mijn stage wil ik dit eerder in mijn proces meenemen, door al tijdens het opzetten van een structuur kort te overleggen met mijn collega's, in plaats van pas achteraf te merken dat wat ik doe niet handig is.

Wat ik daarnaast interessant vind, is de combinatie van mijn achtergrond als programmeur met de meer gebruikersgerichte manier van werken die ik bij Human Centered Design heb geleerd. Ik wil deze twee kanten meer met elkaar gaan verbinden, bijvoorbeeld door in mijn carrière we wat meer te gaan richten op full-stack development.
