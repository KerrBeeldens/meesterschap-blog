We gebruiken ze elke dag, maar hoe werkt een browser eigenlijk van binnen? En waarom moeten browsers alles wat ze ooit ondersteund hebben voor altijd blijven ondersteunen? Deze sessie gaf een helder overzicht van hoe browsers zijn opgebouwd en waar ze vandaan komen.

Geschreven door Kerr

Hoe is een browser opgebouwd?

Een browser bestaat uit een aantal onderdelen die samenwerken. HTML wordt doorgestuurd naar een parser die de DOM opbouwt. CSS gaat naar de CSS parser, die gebruikt wordt door de style engine. De DOM zet je HTML-document om in stukken die JavaScript kan gebruiken. De AOM doet hetzelfde, maar dan voor accessibility.

De style engine gebruikt de DOM om een layout te maken. Dit is de meest kostbare stap in het hele proces. De rendering engine is verantwoordelijk voor het parsen van HTML en CSS, en het opbouwen van de DOM en AOM. In feite is dat alles behalve de JavaScript engine en de interface van de browser zelf. De meest gebruikte rendering engine is Blink, die uit Chromium komt.

Iets om rekening mee te houden is render blocking. Dit houdt in dat het laden van CSS of JavaScript het renderen van de pagina tegenhoudt. Voor JavaScript los je dat op met defer (doe het later) of async (doe het wanneer je wilt, maar niet nu). Voor CSS is dat lastiger. Voor JavaScript is render blocking ongewenst, voor CSS is het in de meeste gevallen juist nodig.

Backwards compatibility

Een belangrijk principe van browsers is dat alles wat ooit ondersteund werd, voor altijd ondersteund moet blijven. Dat klinkt logisch, maar het maakt browsers ingewikkeld om voor te bouwen.

Browsers are the most hostile development platforms in the world.

Als front-end developer moet je rekening houden met een enorme variatie aan devices, schermformaten, browserversies en inputmethodes. Dat is een uitdaging die nooit helemaal weggaat.

Een korte browsergeschiedenis

De allereerste browser was de WWW Browser, gebouwd door Tim Berners-Lee in 1990 voor communicatie binnen CERN. In 1993 kwam Mosaic, de eerste browser die ook voor gewone mensen te gebruiken was. Een jaar later verscheen Netscape als eerste commerciële browser. Internet Explorer volgde in 1995 en was gratis, waardoor het marktaandeel van Netscape snel instortte. Opera kwam in datzelfde jaar uit.

In 2000 verscheen Konqueror, die later de basis zou vormen voor Safari in 2003. Safari baseerde zijn rendering engine WebKit op Konqueror. Firefox kwam uit in 2004. In 2007 bracht Apple Safari uit voor iOS. Chrome volgde in 2008 en splitste zijn rendering engine af van WebKit; dat werd Blink.

Twee recentere browsers die hun eigen rendering engine gebruiken zijn Flow (2020) en Ladybird (2024). Igalia is ook een interessante speler in dit landschap als open-source bedrijf dat bijdraagt aan meerdere rendering engines.

Wat neem ik hiervan mee?

Ik wist wel dat browsers complex zijn, maar had nooit goed nagedacht over wat er precies onder de motorkap gebeurt. Het onderscheid tussen de rendering engine en de JavaScript engine was nieuw voor mij. En dat de JavaScript engine strikt genomen geen onderdeel is van de browser zelf vond ik verrassend.

Het backwards compatibility-principe legt ook iets bloot. Browsers kunnen eigenlijk nooit iets weggooien. Elke keuze die ooit gemaakt is blijft voor altijd meegedragen worden. Dat geeft een ander perspectief op waarom het web soms aanvoelt als een opeenstapeling van laagjes.

Alles wat een browser ooit heeft ondersteund, moet voor altijd blijven werken.

Als front-end developer bouw je voor dat platform. Dat vraagt om begrip van hoe browsers werken, niet alleen van hoe je code schrijft.
