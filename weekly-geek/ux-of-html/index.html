UX of HTML

In deze opdracht ging het om iets wat op het eerste gezicht simpel lijkt: een tekstveld. De uitdaging was om een input-achtig element te bouwen met HTML, CSS en JavaScript, maar zonder het <input> element zelf te gebruiken.

Dat dwingt je om stil te staan bij hoeveel functionaliteit de browser eigenlijk gratis voor je regelt en hoeveel werk het is om dat zelf te repliceren.

Geschreven door Kerr

Wat een input standaard al kan

Het <input type="text"> element is in de basis een vrij eenvoudig ogend veld, maar het doet veel meer dan alleen tekst tonen.

Basisfunctionaliteit

  • Cursorbeheer (invoegen, selecteren, navigeren)
  • Tekstinvoer via toetsenbord
  • Waarde uitlezen via .value in JavaScript
  • Focus- en blur-gedrag

Daarnaast past de browser automatisch gedrag aan, zoals:

  • Virtuele toetsenborden op mobiel
  • Autocomplete ondersteuning
  • Correcte tab-navigatie

Validatie die je gratis krijgt

Een belangrijk onderdeel van inputs is validatie. Zonder extra code krijg je al:

  • required
  • minlength / maxlength
  • pattern
  • readonly
  • disabled
  • size

En daarnaast browsergedrag zoals:

  • :valid en :invalid pseudo classes
  • Standaard foutmeldingen bij form submit
  • Integratie met constraint validation API

Voor specifieke types zoals email komt daar nog extra logica bij:

  • Basis email-validatie (inclusief multiple mode)
  • Automatische toetsenbordoptimalisatie op mobiel
  • Ingebouwde parsing van email-structuren

ARIA textbox: wat je zelf moet overnemen

Als je geen <input> gebruikt en zelf een "textbox" bouwt met bijvoorbeeld <div contenteditable>, dan moet je zelf bijna alles opnieuw implementeren. Met ARIA kun je een element toegankelijk maken via:

<div role="textbox" contenteditable="true"></div>

Maar daarmee ben je er nog lang niet. Je moet zelf zorgen voor:

  • Toegankelijkheidsboom registratie (screenreaders herkennen het als input)
  • Labeling via aria-labelledby
  • Placeholder gedrag via aria-placeholder
  • Multiline gedrag via aria-multiline
  • Required state via aria-required
  • Focus management
  • Keyboard interacties (pijltjes, selectie, delete gedrag)

Wat de browser normaal voor je regelt, moet je nu zelf simuleren.

Wat je zelf allemaal moet bouwen (als je de input niet gebruikt)

Als je het <input> element weglaat, moet je zelf onder andere implementeren:

  • Cursor positionering
  • Text selection (incl. shift + arrow gedrag)
  • Copy/paste ondersteuning
  • IME input (belangrijk voor niet-Latijnse talen)
  • Mobile keyboard gedrag
  • Accessibility tree integratie
  • Form submission integratie
  • Validatie states (valid, invalid, dirty, etc.)
  • Focus ring gedrag
  • Autofill / autocomplete logica

Kort gezegd: je bouwt een klein stukje browser opnieuw.

Waarom dit een slecht idee is

De belangrijkste conclusie van deze oefening is niet dat het onmogelijk is, maar dat het bijna altijd onnodig is.

Browsers hebben tientallen jaren werk gestoken in het oplossen van precies dit probleem. Het <input> element is niet simpel, het lijkt simpel omdat al die complexiteit al is weggeabstraheerd.

Door het zelf te bouwen:

  • verlies je automatisch toegankelijkheid
  • verlies je consistente browsergedrag
  • moet je edge cases zelf oplossen
  • introduceer je snel bugs in input-gedrag

En dat alles voor iets dat al native bestaat.

Wat ik hiervan heb geleerd

Wat deze oefening vooral duidelijk maakte, is hoeveel verantwoordelijkheid de browser eigenlijk overneemt.

Een inputveld is niet alleen een visueel element, maar een combinatie van toegankelijkheid, validatie interactie en platformgedrag.

Ik heb geleerd dat het vrijwel altijd beter is om bestaande HTML elementen te gebruiken in plaats van ze na te bouwen. Niet omdat het niet kan, maar omdat je nooit alle randgevallen even goed afdekt als de browser dat doet.

De belangrijkste les is eigenlijk simpel:

Als de browser iets al oplost, is het bijna altijd beter om dat te gebruiken dan het opnieuw te proberen bouwen.
