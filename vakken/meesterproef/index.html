Onderzoek naar te gebruiken technieken

Binnen de groep hebben we afgesproken om voor de overgebleven dagen vooral goed onderzoek te doen naar hoe we de tool gaan opzetten. Dit betekent dat iedereen aan de slag is gegaan met CodePens, schetsen voor de user interface en onderzoek naar bruikbare libraries en web API's.

Ik heb vooral gekeken naar de Canvas web API. Ik heb met deze API een opzet gemaakt voor hoe de animatietool zou kunnen werken in code. Het doel hiervan was tweevoudig:

  • Experimenteren en zoeken naar een logische code structuur om animaties op te zetten

Uiteindelijk leidden deze experimenten tot de volgende inzichten:

  • De code structuur die het best lijkt te gaan werken voor dit project is om de code op te splitsen in een user interface, data met betrekking tot de animatie en een tussenlaag hiertussen die een vertaalslag maakt tussen de animatiedata en de tool.

Uiteindelijk kwam Sabrina met het idee om de library GSAP te gebruiken. Deze library zag er veelbelovend uit, dus zijn verschillende groepsleden begonnen hier verder naar te kijken. Hier kwam uiteindelijk een redelijk werkend prototype uit, dus hebben we besloten om hier voor nu verder mee te werken.

Bedenken opbouw animatiedata

Op dit punt hadden we als groep een werkend prototype, maar de structuur van de code was erg rommelig en niet makkelijk uitbreidbaar. Ook is een belangrijke eis van Marjolijn dat de animatie uiteindelijk opgeslagen kan worden. Ik had daarom de suggestie gedaan om de animatie als een object op te bouwen en deze uiteindelijk in JSON-formaat op te slaan en weer in te laden. Ik heb hierin het voortouw genomen en heb met de groep de volgende structuur bedacht:

  • Algemene metadata (name, duration)
  • Elementen (elements)
  • Groepen (groups)
  • Animaties per element (animations)

Dit ziet er bijvoorbeeld zo uit:

{
  "name": "Draaiende Titel",
  "duration": 5,
  "elements": {
    "title": "Welkom"
  },
  "groups": {
    "main": {
      "groupName": "Hoofdgroep",
      "members": [
        "title"
      ]
    }
  },
  "animations": {
    "title": {
      "scaleY": [
        {
          "progress": 0,
          "value": 1,
          "ease": "none"
        },
        {
          "progress": 1,
          "value": 2,
          "ease": "none"
        }
      ],
      "rotation": [
        {
          "progress": 0,
          "value": 0,
          "ease": "none"
        },
        {
          "progress": 1,
          "value": 360,
          "ease": "none"
        }
      ]
    }
  }
}

In dit voorbeeld:

  • De animatie duurt 5 seconden.
  • Er is één element: "Welkom".
  • Het element zit in de groep "Hoofdgroep".
  • Tijdens de animatie draait het element één volledige rotatie.
  • Tegelijkertijd wordt de hoogte (scaleY) verdubbeld.

Ik heb afgesproken met de groep dat ik volgende week deze structuur in de animatietool ga verwerken.

Week 2: Opzetten basiscode

De tweede week stond in het teken van het opzetten van de basiscode van het project. We hadden op dit punt als groep besloten dat we GSAP zouden gebruiken en zoals ik bij week 1 had benoemd, hebben we ook een opzet van de datastructuur in JSON-formaat. Ik had het op me genomen om ervoor te zorgen dat deze datastructuur omgezet zou worden in een logische code architectuur.

Code architectuur

Ik ben vooral bekend met object georiënteerde talen, dus ik heb de keuze gemaakt om een klasse op te zetten die simpelweg AnimationBuilder heet. De klasse wordt door de UI aangeroepen om keyframes te kunnen toevoegen en de animatie te bouwen en vervolgens af te spelen. De klasse heeft daarnaast ook de mogelijkheid om het resultaat hiervan om te zetten naar JSON-formaat zodat de animatie opgeslagen kan worden, en ook om een animatie in te laden op basis van een JSON-object.

In hoofdlijnen heeft de AnimationBuilder de volgende functies:

  • buildAnimation(), die verantwoordelijk is voor het opbouwen van de animatie op basis van de keyframes die op de animatie staan.
  • setKeyframe(), een functie om keyframes aan de animatie toe te voegen.
  • play(), pause() en setProgress() en enkele andere functies om de animatie te kunnen afspelen.
  • loadAnimation() en saveAnimation() om animaties op te kunnen slaan en in te laden.

Ik ben me ervan bewust dat dit nu erg veel verantwoordelijkheden zijn voor een enkele klasse, dus daarom ben ik van plan om deze verantwoordelijkheden verder op te splitsen.

Werken met GSAP

Ik heb hiervoor leren werken met GSAP. Dit is een library die gebruikt kan worden om eenvoudig CSS-properties te animeren. Ten eerste maak ik een Timeline-object aan met:

gsap.timeline({
	paused: true,
	repeat: -1,
	onUpdate: () => {this.onUpdateListener(this.timeline)},
});

Ik geef hierbij aan dat de timeline standaard op pauze moet staan, oneindig vaak loopt en ik voeg een onUpdateListener toe. Dit is een callback die wordt aangeroepen als de timeline wordt aangepast.

Vervolgens itereer ik over een object van keyframes om ervoor te zorgen dat er keyframes aan de timeline worden toegevoegd. Dit kan met de volgende code:

this.timeline.to(
	`.${target}`,
	{
		[property]: current.value,
		duration: timeDifferenceSeconds,
		ease: current.ease ? current.ease : "none",
	},
	last.progress * this.durationSeconds,
);

In deze code is target het element waarop de keyframe moet worden gezet, current is de huidige keyframe, last is de vorige keyframe (nodig om ervoor te zorgen dat de animatie op het juiste moment begint) en timeDifferenceSeconds is de tijd in seconden tussen de current en last keyframe.

In het kort zorgt timeline.to() ervoor dat op een specifiek element een keyframe wordt gezet, op een specifieke property. De to()-functie houdt in dat de property een bepaalde startwaarde heeft en naar een nieuwe waarde moet worden geanimeerd. Dit gebeurt voor een bepaalde duration en volgens een specifieke ease-functie. De laatste parameter is het tijdstip waarop de animatie moet beginnen met spelen; dat is het moment waarop een vorige keyframe gezet is, vandaar dat last nodig is.

Het was hierbij van belang dat de keyframes op volgorde stonden. Een handigheidje dat ik hierbij geleerd heb is de Array.sort()-functie van JavaScript:

keyframes.sort((a, b) => a.progress - b.progress);

keyframes is hierbij de lijst van keyframes op een property. Elke keyframe heeft een progress-waarde, een getal tussen 0 en 1, waarbij 0 het begin van de animatie is en 1 het einde. Als een keyframe dus een kleinere progress heeft dan een andere keyframe, komt die keyframe eerst. In de Array.sort()-functie wordt dit als volgt bepaald: men heeft een keyframe a en een keyframe b die met elkaar vergeleken worden, en de functie verwacht dat een getal wordt teruggegeven:

  • Een negatief getal geeft aan dat a voor b komt
  • Een positief getal geeft aan dat b voor a komt
  • 0 geeft aan dat a en b gelijk aan elkaar zijn

a.progress - b.progress is dus precies wat je nodig hebt: als de progress van a voor b komt (en dus kleiner is), dan is het resultaat negatief en weet JavaScript dat a voor b moet komen. Deze functie geeft dan uiteindelijk een gesorteerde lijst van keyframes terug.

Week 3: Meters maken

De derde week was wat turbulent. In het begin van de week was ik rustig aan het werken om de code architectuur te verbeteren en grotendeels af te maken, maar in het tweede gedeelte van de week ben ik vooral druk bezig geweest om de code leesbaarder te maken voor mijn groepsgenoten.

Verbeteren code

Vorige week had ik al aangegeven dat mijn code nog vrij rommelig was. De AnimationBuilder is op dit moment een vrij complexe klasse met verschillende verantwoordelijkheden. Dit heb ik in eerste instantie bewust zo gedaan, omdat het een valkuil van mij is om direct vanaf het begin een ingewikkelde code architectuur op te zetten die achteraf gezien niet handig blijkt te zijn, of soms wel handig maar overkill voor het project. Daarnaast moet ik natuurlijk ook rekening houden met de leesbaarheid voor mijn groepsgenoten. Op dit punt was echter duidelijk dat de code verder opgedeeld moest worden.

De AnimationBuilder had op dit punt globaal de volgende verantwoordelijkheden:

  • Het opslaan/inladen van de animatie
  • Het bijhouden van de data van de animatie
  • Het opbouwen van de animatie
  • Het afspelen van de animatie

Ik denk dat idealiter elk van deze aspecten zijn eigen klasse zou moeten zijn, maar ik heb ervoor gekozen, vooral met oog op simpliciteit, om de code op te delen in twee stukken:

  • Een AnimationData-klasse die de data van de animatie bevat en ervoor zorgt dat deze kan worden opgeslagen en ingeladen vanuit JSON.
  • Een AnimationPlayer-klasse die de animatie opbouwt en verantwoordelijk is voor het afspelen ervan.

Het fijne van deze architectuur is dat de dependency van GSAP, die eerst nog in de AnimationBuilder zat, verplaatst kan worden naar de AnimationPlayer. Op deze manier kan de AnimationData-klasse bestaan zonder te weten welke library er gebruikt wordt om de animatie op te bouwen. De AnimationData wordt meegegeven aan de AnimationPlayer, wat leidt tot een soort dependency inversion: GSAP wordt nu afhankelijk van de domeinklasse, in plaats van andersom.

Ik was zeer tevreden met deze opsplitsing, maar had hierbij niet goed de complexiteit van de code in de gaten gehouden. Tijdens een code review met Cyd kwam ik erachter dat de rest van de groep nog niet mee was met de opbouw van de code, en dat dit de projectvoortgang in gevaar bracht.

Ik heb daarom het tweede gedeelte van de week hard gewerkt om ervoor te zorgen dat de code duidelijk was voor groepsgenoten. Dit heb ik gedaan door eerst samen met Mila de code door te spitten: zij stelde vragen over de code die ik probeerde te beantwoorden, en zij schreef in haar eigen woorden comments die de code uitlegden. Dit zorgde er niet alleen voor dat de code duidelijker werd voor de groep, maar ook dat kleine foutjes of onhandigheden aan het licht kwamen.

In sommige gevallen was het nodig om de JavaScript-kennis van de groep bij te spijkeren; zo vond ik het van belang dat er gewerkt werd met een Map, omdat dit praktisch was voor de code architectuur en de code alleen maar ingewikkelder zou worden als dat een array zou zijn. Maar er waren ook plekken in de code die door deze review onnodig complex bleken te zijn. Zo had ik op verschillende plaatsen een getter die, in plaats van de originele map, een object teruggaf, zoals hieronder:

getElements() {
return structuredClone(Object.fromEntries(this.#elements));
}

Vervolgens werd dit ergens anders in de code op de volgende manier gebruikt:

Object.entries(elements).forEach(([id]) => {targetIds.push(id);});

De functies Object.fromEntries() en Object.entries() zijn vrij omslachtig en lezen moeilijk. Op het moment dat ik dit schrijf, merk ik zelf ook te twijfelen wat de functies ook alweer precies doen, wat aangeeft dat de code beter kan.

Het is veel makkelijker om de map direct terug te geven:

getElements() {
	return structuredClone(this.#elements);
}

De eerdere code wordt dan simpelweg:

elements.keys().forEach(id => targetIds.push(id));

Uiteraard moeten groepsgenoten dan nog steeds begrijpen dat Map.keys() een iterator teruggeeft voor alle keys van een map, maar zelfs als dit niet in een keer duidelijk is, is de code een stuk leesbaarder.

Verder heb ik ook gewerkt met JSDoc. Hiermee kan op de volgende manier documentatie worden toegevoegd aan JS:

/**
* @description Create a new element in the animation
* @param {string} elementName The display name of the new element
* @returns {string} the internal UUID of the element used for referencing
*/

Met @description kan een omschrijving worden toegevoegd aan een functie, met @param {type} naam kan een parameter worden omschreven en het type worden aangegeven, en met @returns {type} kan worden aangegeven wat de functie teruggeeft en van welk type die waarde is.

Dit bleek veel waarde te hebben voor dit project, omdat groepsgenoten nu een beter beeld hadden van wat een functie deed voordat ze de code begonnen met lezen, waardoor ze de code ook sneller konden begrijpen.

Week 4: Afwerken functionaliteiten

De vierde week stond in het teken van het afronden van de belangrijkste functionaliteiten, namelijk de save/load-functionaliteit en het implementeren van de undo- en redo-knoppen. Gelukkig zat dit door de eerder gemaakte code architectuur in hoofdlijnen best mee.

Save/load

Deze week heb ik gewerkt aan de save- en loadfunctionaliteit van de animatietool. Omdat ik in de voorgaande weken veel tijd had besteed aan het opzetten van een goede datastructuur (AnimationData) en het kunnen serialiseren van deze data naar JSON, bleek dit achteraf een goede investering te zijn. Het opslaan van een project bestond daardoor grotendeels uit het omzetten van de animatiedata naar JSON en deze vervolgens aan te bieden als download.

Wel wilde ik dat een opgeslagen animatie ook echt zelfstandig gebruikt kon worden, zonder afhankelijk te zijn van de editor. Daarom heb ik ervoor gekozen om niet alleen de JSON op te slaan, maar een volledig HTML-bestand te genereren waarin de animatie en alle benodigde data zijn ingebed. Hierdoor kan een gebruiker een animatie openen door simpelweg het HTML-bestand in een browser te openen.

Het genereren van zo'n HTML-bestand was een interessante uitdaging. De complete HTML, CSS en JavaScript worden als een template string opgebouwd, waarna de JSON-data rechtstreeks in het document wordt geplaatst. Hierdoor hoeft het bestand geen externe JSON meer op te halen, wat belangrijk is omdat een lokaal geopend HTML-bestand (file://) geen toegang heeft tot fetch() zonder een webserver.

Daarnaast heb ik ervoor gezorgd dat ook instellingen zoals het gekozen thema worden opgeslagen. Bij het genereren van de HTML worden de juiste CSS-variabelen direct in het document geplaatst, zodat de animatie er hetzelfde uitziet als tijdens het ontwerpen.

Voor het inladen van een project kon ik gebruikmaken van de eerder gemaakte fromJSON()-functie binnen de AnimationData-klasse. Nadat een bestand is geselecteerd, wordt de JSON ingelezen met een FileReader en wordt de complete animatie opnieuw opgebouwd. Vervolgens worden de interface en preview opnieuw gesynchroniseerd met de geladen data. Hierbij heb ik er bewust voor gekozen om ook de undo/redo-geschiedenis leeg te maken en direct een nieuw beginsnapshot aan te maken. Hiermee voorkom ik dat een gebruiker na het openen van een nieuw project nog kan teruggaan naar een volledig andere animatie.

Achteraf ben ik tevreden met deze aanpak. Doordat de data en de speler al los van elkaar waren ontworpen, hoefde de save/load-functionaliteit nauwelijks rekening te houden met de interne werking van GSAP. Het enige wat opgeslagen hoeft te worden is de animatiedata; de animatie zelf wordt bij het openen opnieuw opgebouwd.

Undo/redo

Naast de save/load-functionaliteit heb ik deze week de undo- en redofunctionaliteit toegevoegd. Dit bleek een mooie toepassing van het Memento Pattern, een ontwerppatroon dat bedoeld is om eenvoudig eerdere toestanden van een object te kunnen herstellen zonder dat de rest van de applicatie weet hoe deze toestand intern is opgebouwd.

Ik heb hiervoor een aparte History-klasse gemaakt die een lijst van snapshots (mementos) bewaart. Elke snapshot is een volledige kopie van de huidige AnimationData. Om te voorkomen dat een wijziging achteraf ook oude snapshots zou aanpassen, worden alle mementos opgeslagen met behulp van structuredClone(). Hierdoor bevat iedere snapshot een volledig onafhankelijke kopie van de animatie.

De klasse houdt daarnaast een currentIndex bij die aangeeft welke snapshot op dit moment actief is. Wanneer een nieuwe wijziging wordt opgeslagen nadat de gebruiker eerst een aantal keer op undo heeft gedrukt, worden alle toekomstige snapshots verwijderd. Dit gedrag is hetzelfde als in de meeste teksteditors en voorkomt dat er meerdere vertakkingen in de geschiedenis ontstaan.

De implementatie van undo en redo zelf bleef daardoor relatief eenvoudig. De History-klasse hoeft alleen de index naar voren of achteren te verplaatsen en de juiste snapshot terug te geven. Vervolgens wordt deze snapshot weer teruggezet in de AnimationData, waarna de interface en de GSAP-timeline opnieuw worden opgebouwd.

Tijdens het ontwikkelen realiseerde ik me opnieuw hoe belangrijk een goede scheiding tussen data en presentatie is. Doordat alle animatiegegevens zich in één datastructuur bevinden, is undo/redo uiteindelijk niets meer dan het terugzetten van een eerdere versie van die data. Als de UI of GSAP zelf de bron van waarheid was geweest, zou deze functionaliteit aanzienlijk ingewikkelder zijn geworden.

Ik ben tevreden met hoe deze functionaliteit uiteindelijk is geworden. De combinatie van snapshots en het Memento Pattern levert een relatief eenvoudige, maar betrouwbare oplossing op die bovendien goed uitbreidbaar blijft wanneer er later nieuwe onderdelen aan de animatiedata worden toegevoegd.
